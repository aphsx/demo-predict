Section 06

Why These Choices?

Every engineering decision vs the alternatives.

Arq + Redis
vs
Kafka
Kafka handles millions of events/second and needs 3+ brokers to run. You have 5 users running ~10 jobs/day. Arq + Redis is already running and handles thousands of jobs/second.
Elysia (TypeScript)
+
FastAPI (Python)
LightGBM, SHAP, BG-NBD, Optuna only exist in Python. But modern auth and typed APIs are better in TypeScript. Both services exist because neither language can replace the other here.
SQL rows in PG
vs
Files in R2
SHAP explain needs to load one customer's full history. With SQL that's a 3ms indexed query. With R2 you'd download a 50MB Excel just to get one person's data.
SSE + Redis Streams
vs
WebSockets
Progress updates are server-push only — the browser never sends data back. SSE is built for this: auto-reconnect, works through any proxy, simpler than a full bidirectional WebSocket.
LightGBM
vs
Neural Network
Gradient boosted trees beat neural networks on tabular business data at this scale (25k rows, 30 features). LightGBM trains in seconds, not hours, and works with SHAP for interpretability.
Drizzle (introspect)
vs
Drizzle (generate)
Alembic already owns the schema. Drizzle reads (introspects) what Alembic built — it never generates migrations. One source of truth for schema: Alembic. One ORM per runtime.
